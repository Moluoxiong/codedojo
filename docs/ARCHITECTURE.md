## 1. 团队组织结构与 AI 角色定义

为了实现产品成功与团队成长双重目标，本项目采用了融入 AI 助手的混合组织结构。

### 1.1 核心团队角色与职责

| 角色名称 (Role) | 投入时间 | 核心职责 (Core Responsibilities) | 重点学习/关注点 |
| :--- | :--- | :--- | :--- |
| **Edward Ma** (开发负责人/主要贡献者) | 70% | **执行与流程管理：** 主导编码实现；学习并执行工程流程。 | 解决实际编码问题，寻求技术解释。 |
| **Jim Ma** (初级工程师/流程学习者) | 20% | **流程学习与辅助实现：** 学习流程和规范；实现辅助任务。 | 流程指导、简单任务的实现方法。 |
| **Jackie Ma** (技术导师/架构师) | 极少 | **教练与决策：** 关键技术决策；高层 Code Review；解决核心技术难题。 | 获取架构建议和技术风险分析。 |

### 1.2 AI 协作角色定义（创新实践）

本项目引入两个 AI 角色，负责知识管理和流程推进。

| AI 角色 (AI Role) | 核心职责 (Core Responsibilities) | 目的 |
| :--- | :--- | :--- |
| **AI 新手教练** | **知识整理与学习指导：** 整理 Edward/Jim 在编码中产生的技术知识点和决策，为 **新手历程** 准备内容。 | 确保团队的经验和知识被有效记录和传承。 |
| **AI 项目经理** | **流程推进与文档管理：** 提醒 Edward/Jim 哪些对话内容应归档；将项目决策转化为 **GitHub 任务卡和文档格式**。 | 保证项目流程的规范性和文档的完整性。 |

## 2. GitHub 基础设施与文档结构

### 2.1 版本管理基础设置
* **GitHub 组织:** 统一使用组织账号（如 `CodeDojo-Edu`），方便权限管理。(not use now)
* **仓库命名:** `codedojo` (私有仓库)。
* **Branching Policy:** 遵循简化 TBD 流程：`main` $\leftarrow$ `develop` $\leftarrow$ `feature/xxx`。
* **流程保障:** `main` 分支设置保护规则：**必须通过 Code Review 才能合并**。

### 2.2 Monorepo 目录结构

采用 Monorepo 结构分离前端、后端和文档，便于管理 P1, P2, P3 阶段的不同组件。
